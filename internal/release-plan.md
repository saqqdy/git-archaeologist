## v0.1.0 — 数据基座 (Week 1)

> **用户故事**：作为开发者，我可以用一条命令获取文件/行的完整 blame 信息，而不需要手写 `git blame --line-porcelain` 再人工解析

### 功能清单

| # | 功能 | 优先级 | 验收标准 |
|---|------|--------|----------|
| 1 | `git blame --line-porcelain` 封装 | P0 | 正确解析 100 行文件 <1s |
| 2 | `git log --format=json` 封装 | P0 | 支持时间范围、作者、路径过滤 |
| 3 | `git diff <commit>` 封装 | P0 | 正确解析 hunk 头 + 内容 |
| 4 | `git show <commit>` 封装 | P1 | 输出完整的 commit 内容 |
| 5 | `git log --follow` 重命名跟踪 | P1 | 跟踪至少 2 次重命名 |
| 6 | 三级缓存 (session → fs → project) | P1 | 重复查询命中 >50% |
| 7 | TypeScript 类型定义 | P0 | 所有数据结构有完整类型 |
| 8 | 错误处理 (非法路径、空仓库等) | P1 | 不抛出未捕获异常 |
| 9 | 日志/调试模式 | P2 | `DEBUG=git-arch:*` 可查看命令详情 |

### 交付物

- `@git-archaeologist/core` npm 包
- API 文档 (TSDoc)
- 基础测试套件 (>80% 覆盖率)

### 不包含

- AI 分析 (v0.2+)
- GitHub/GitLab 集成 (v0.3+)
- CLI 命令 (v0.4+)

## v0.2.0 — 语义分析 (Week 2)

> **用户故事**：作为开发者，我可以看到自动分组的 commit 列表和意图标签，快速理解"这批变更是修 bug 还是加功能"

### 功能清单

| # | 功能 | 优先级 | 验收标准 |
|---|------|--------|----------|
| 1 | Commit 分组 (merge/PR/时间窗口/分支) | P0 | 非 merge 情况下每组 <=5 commit |
| 2 | 意图分类器 (feat/fix/refactor/perf/workaround/revert/deps) | P0 | 准确率 >85% |
| 3 | 变更热点检测 | P1 | 识别 Top 10 高频文件 |
| 4 | 作者专长映射 | P1 | 按 commit 数量 + 变更范围推断 |
| 5 | 演变时间线构建 | P0 | 输出有序 + 标注的 ChangeGroup[] |
| 6 | 分析结果可序列化 | P1 | 支持 JSON 输出供下游消费 |

### 交付物

- `@git-archaeologist/analyzer` npm 包
- 意图分类基准测试 (3 个已知仓库)
- 分析 API 文档

## v0.4.0 — AI 解释 (Week 4) — MVP

> **用户故事**：作为开发者，我可以用 `/why src/auth.ts:42` 获得"这段代码为什么写成这样"的人话解释

### 功能清单

| # | 功能 | 优先级 | 验收标准 |
|---|------|--------|----------|
| 1 | `/why <file:line>` 命令 | P0 | 输出含证据链的因果解释 |
| 2 | `/history <path\|fn>` 命令 | P0 | 输出演变时间线 + 关键决策 |
| 3 | `/context <snippet>` 命令 | P1 | 输出关联变更网络 |
| 4 | `/decisions [area]` 命令 | P1 | 输出类 ADR 格式决策记录 |
| 5 | 置信度评分 (🟢🟡🔴) | P0 | 100% 解释附带置信度 |
| 6 | 证据链溯源 (commit→PR→Issue) | P0 | 每个结论可追溯到原始数据 |
| 7 | Claude Code Skill 注册 | P0 | 命令可在 Claude Code 中直接调用 |

### 交付物

- Claude Code Skill 包 (可安装)
- 4 个命令的使用示例
- 与人工对照的准确率报告

### 不包含

- `/archaeology` 交互模式 (v1.0.0)
- GitLab 集成 (v1.1.0)

### MVP 验证指标

| 指标 | 目标 | 度量方式 |
|------|------|----------|
| 解释准确率 | >80% | 人工比对 50 个 `/why` 结果 |
| 解释用时 | <10s | 单次命令端到端计时 |
| 用户满意度 | >7/10 | 5 人内测打分 |
| 致命 bug 数 | 0 | 内测期间零崩溃 |

## 版本号规则

```
MAJOR.MINOR.PATCH

MAJOR: 不兼容的 API 变更
MINOR: 向后兼容的新功能
PATCH: 向后兼容的 bug 修复

v0.x.y: 初始开发阶段，API 可能随时变更
v1.0.0: 第一个稳定版本，承诺 API 兼容性
```

### 分支策略

```
main ───────────────────────────────────────────→ 稳定发布
  │
  ├── develop ──────────────────────────────────→ 集成分支
  │     │
  │     ├── feature/blame-parser ───→ 合并到 develop
  │     ├── feature/intent-classify ─→ 合并到 develop
  │     └── feature/github-integration → 合并到 develop
  │
  ├── release/v0.4.0 ──→ 从 develop 切出 → 测试 → 合并到 main
  └── hotfix/xxx ──→ 从 main 切出 → 修复 → 合并回 main + develop
```

### 发布流程

```
1. 从 develop 切出 release/vX.Y.Z 分支
2. 更新 CHANGELOG.md、package.json 版本号
3. 运行完整测试套件 + E2E 测试
4. 在 3 个真实仓库上做 smoke test
5. 合并到 main，打 tag vX.Y.Z
6. npm publish
7. 合并回 develop
8. 更新 GitHub Release Notes
```
